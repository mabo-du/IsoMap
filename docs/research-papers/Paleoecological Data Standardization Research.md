# **Architectural and Standardisation Frameworks for Paleoecological and Isotopic Data Repositories**

## **Introduction to Data Integration in Paleoecology and Archaeology**

The ongoing digitisation of the paleoecological, archaeological, and paleoclimatological sciences has catalysed a profound shift from siloed, bespoke research datasets towards integrated, federated data ecosystems. As the sheer volume of multi-proxy data—spanning pollen counts, vertebrate remains, diatom assemblages, and stable isotopic measurements—continues to expand at an unprecedented rate, the necessity for robust cyberinfrastructure becomes paramount. The central challenge in this domain is the standardisation of legacy research spreadsheets. Historically, these datasets have been characterised by idiosyncratic column naming conventions, heterogeneous taxonomic classifications, incompatible depth-age models, and highly inconsistent reporting of analytical metadata. To align these legacy datasets with the Findable, Accessible, Interoperable, and Reusable (FAIR) data principles, as well as the Collective Benefit, Authority to Control, Responsibility, and Ethics (CARE) guidelines, a comprehensive and automated standardisation architecture is urgently required.1  
The development of an open-source standardisation tool to ingest, clean, and map these legacy spreadsheets necessitates a highly multifaceted approach. It requires the integration of rigorous schema mapping algorithms, programmatic interactions with established domain repositories (such as the Neotoma Paleoecology Database, IsoArcH, and PANGAEA), real-time taxonomic validation against global nomenclatural authorities, and the generation of reproducible provenance trails. This comprehensive research report provides an exhaustive, expert-level analysis of the requisite technical workflows, database structures, application programming interfaces (APIs), and reporting standards necessary to architect a state-of-the-art data standardisation pipeline for the paleo-geosciences.

## **The Neotoma Paleoecology Database: Submission Workflows and Data Architecture**

The Neotoma Paleoecology Database represents an international, community-led coalition of constituent databases providing a comprehensive, open-access repository for Pliocene, Pleistocene, and Holocene fossil data.2 Operating under a distributed scientific governance model, Neotoma relies on Data Stewards from various geographic regions and taxonomic disciplines to coordinate, manage, and vet data prior to its permanent archival.2

### **Hierarchical Data Structure and Core Tables**

The fundamental architectural design of Neotoma relies on a highly relational PostgreSQL database, currently operating on version 15.12, which comprises 181 tables, 6 views, and 1,074 distinct columns.4 The schema enforces a strict spatial and temporal hierarchical relationship between geographic locations and the specific analytical measurements derived from them. The primary relational hierarchy flows downward from spatial metadata to the atomic observational data.4

| Neotoma Table | Record Volume (approx.) | Architectural Function and Data Content |
| :---- | :---- | :---- |
| Sites | System-wide | Contains core spatial metadata, including latitude, longitude, and site name.4 |
| CollectionUnits | 44,136 | Represents the specific physical extraction, such as a sediment core, an archaeological excavation trench, or a packrat midden. It links directly to the parent Site.4 |
| AnalysisUnits | 878,071 | Defines the specific stratigraphic depths, horizons, or discrete intervals within a CollectionUnit where physical samples were isolated.4 |
| Chronologies | 43,277 | Stores explicit timelines assigned to a CollectionUnit. A single core may possess multiple chronologies (e.g., raw radiocarbon years vs. calibrated calendar years).4 |
| ChronControls | 400,744 | The age-depth control points used to generate age models, such as specific radiocarbon dates, volcanic tephra layers, or biostratigraphic markers.4 |
| Data | 11,805,526 | The primary, highly normalised atomic data table. Every single occurrence of a proxy variable (e.g., a specific pollen taxon count) constitutes a distinct record.3 |

The structural integrity of this hierarchy dictates the complexity of the data submission process. A legacy spreadsheet, which typically presents data in a flat, two-dimensional matrix (with depths as rows and taxa/isotopes as columns), must be mathematically pivoted and fragmented to populate this highly normalised relational structure.

### **Technical Submission Workflows: Tilia, APIs, and DataBUS**

The traditional, GUI-based method of data curation and ingestion into Neotoma relies on the Tilia software ecosystem. Tilia is a Windows-based desktop application designed explicitly for managing, graphing, and validating stratigraphic microfossil data.5 Tilia interacts with the Neotoma database through RESTful JSON APIs, serving as the primary curation tool for domain-expert Data Stewards. Because Neotoma operates on a strict vetting model, raw data cannot typically be written directly to the production database via a public POST endpoint by an unauthenticated user; rather, it must undergo verification by Data Stewards who assemble .tlx files.2  
For programmatic, bulk, or automated submissions, Neotoma provides the DataBUS infrastructure.7 The DataBUS pipeline circumvents the manual bottleneck by leveraging a temporary "holding tank" database (neotomaholdingtank).4 This allows data to be programmatically pushed via Python scripts, algorithmically verified, reviewed by stewards using Tilia, and finally migrated into the production schema.4

### **Required Fields for Diverse Dataset Types**

Neotoma supports a vast array of observational proxies, including pollen, vertebrate fauna, diatoms, plant macrofossils, testate amoebae, and ostracodes.3 However, the database specifically prohibits the ingestion of derived variables (e.g., reconstructed July mean temperatures or species richness indices) and raw image files.3  
The required fields vary depending on the proxy type, but universally mandate:

1. **Spatial and Contextual Metadata**: Site location, depositional environment (depenvttypes), and collection methodologies.4  
2. **Taxonomic Identifiers**: Validated taxonomic names mapped to Neotoma's internal taxonomic trees.  
3. **Geochronological Data**: Uncalibrated dates (e.g., ![][image1], ![][image2]) and the specific models used to infer age-depth relationships.3  
4. **Raw Observations**: Metrics such as the Number of Identified Specimens (NISP), Minimum Number of Individuals (MNI), presence/absence binary states, or specific isotopic values linked to a specimen ID.3

For specific integrations, such as the incorporation of the SISAL (Speleothem Isotope and Synthesis AnaLysis) database into Neotoma, highly specialised metadata fields are requested. The SISAL submission templates require fields such as Drip type (e.g., Seepage flow, Seasonal drip), Mineralogy (e.g., Calcite, Aragonite), and δ18O water equilibrium.8 Legacy datasets frequently lack this granular metadata; historical analyses show that in previous SISAL iterations, information for Drip type was unknown for 80.9% of records, and δ18O water equilibrium was unknown for 55.9% of records.9 A robust standardisation tool must be capable of handling these high degrees of missingness without failing schema validation.

### **Schema Mapping and YAML Template Structures**

To bridge the gap between arbitrary legacy CSV files and the strict Neotoma relational schema, the DataBUS workflow employs YAML-based crosswalk templates.4 The automated standardisation tool must generate or consume these YAML templates to execute valid submissions.  
The template is defined by an apiVersion header (currently neotoma v2.0) and a kind parameter (typically set to development during the validation phase).4 The core of the template is the metadata array, which provides explicit declarative instructions on how to map each column in the source spreadsheet to the corresponding Neotoma table and column.

| YAML Field | Description and Functionality | Expected Data Types / Values |
| :---- | :---- | :---- |
| column | The exact string literal of the column header in the source legacy spreadsheet. | String |
| neotoma | The target database schema path, combining the table and column name (e.g., ndb.sites.sitename). | String |
| vocab | Specifies whether the field requires validation against a controlled vocabulary list. If applicable, provides the array of acceptable terms (e.g., \`\`). | Array of Strings or False |
| repeat | Indicates if a value is uniquely tied to a specific row (False) or if a single value applies uniformly to the entire dataset/column (True). | Boolean |
| type | The expected PostgreSQL data type to ensure schema compatibility before insertion. | integer, numeric, date, character, string |
| ordered | Defines whether the specific top-to-bottom sequence of the column values possesses analytical significance (e.g., chronological or stratigraphic ordering). | Boolean |

Validation routines applied during the DataBUS upload phase generate highly detailed log reports that verify unit definitions, assert that dating horizons fall within reported depth bounds, and check vocabulary compliance.7 Datasets that fail these programmatic checks are rejected before reaching the holding tank.

## **IsoArcH, IsoMemo, and IsoBank: Isotopic Data Ingestion and Schemas**

While Neotoma handles a broad spectrum of paleoecological proxies, the archiving of isotopic data for bioarchaeology, paleodietary reconstruction, and paleoenvironmental analysis is heavily centralised within the IsoMemo network, and specifically within the IsoArcH repository.1 IsoArcH operates as a non-profit initiative dedicated to preserving isotopic data under FAIR and CARE paradigms, ensuring ethical data usage, particularly concerning human remains.11

### **Submission Requirements and API Limitations**

The ingestion architecture for IsoArcH is fundamentally template-driven, rather than API-driven for raw data uploads. Currently, there is no publicly documented RESTful API endpoint for direct, automated submission of raw datasets to IsoArcH. Instead, submissions are processed via standardised templates available in Microsoft Excel or Grist formats.12 The submission pipeline features a drag-and-drop web interface where files are imported and run through immediate structural validation checks.12 Once a dataset passes this initial validation, it is routed to the IsoArcH curation team for harmonization and the assignment of a persistent Digital Object Identifier (DOI), allowing for immediate citation or the imposition of a temporary embargo period.11  
To support institutional funding mandates, IsoArcH provides an extensive Data Management Plan (DMP) template that outlines the exact protocols for collecting, organising, and preserving isotopic research.13 This DMP template is rooted in the Science Europe core requirements and seamlessly integrates policy frameworks with technical data management.13

### **Data Schema and Controlled Vocabularies**

The data models required by the IsoMemo ecosystem, and analogous systems like IsoBank, rely heavily on controlled vocabularies to prevent the semantic drift that plagues legacy datasets. A standardisation tool must successfully identify and translate idiosyncratic legacy fields into these rigid schema requirements.  
For instance, the IsoBank schema (which overlaps conceptually with IsoMemo) defines explicit attributes for metadata fields. Each field operates under strict "deonticity" (requirement level), conditional usage rules, and predefined controlled vocabularies.15

| Metadata Group | Standardised Field Name | Description | Deonticity (Requirement) |
| :---- | :---- | :---- | :---- |
| Core Analysis | analytical\_lab | The specific laboratory facility where the mass spectrometry or isotopic measurement occurred. | Required |
| Core Analysis | analytical\_matrix | The specific biological, geological, or chemical substance subjected to analysis (e.g., bone collagen, tooth enamel carbonate). | Required |
| Measurement | isotope\_system | The specific elemental isotope ratio being reported (e.g., ![][image3], ![][image4], ![][image5]). | Required |
| QA/QC | collagen\_yield | The percentage of collagen successfully extracted from the raw bone or dentine mass, used to identify diagenesis. | Conditional / Recommended |

Submissions to IsoArcH also require detailed site-level information, chronology, and taxonomic identification of the species sampled.17 The integration of archaeological context with the biogeochemical data ensures that isotopic values can be subsequently queried across temporal and spatial dimensions via the IsoMemo and Pandora data platforms.18 For example, studies leveraging this database infrastructure have successfully mapped the spatial and dietary niches of Late Pleistocene hunter-gatherer prey species—such as *Rangifer*, *Equus*, and *Bovinae*—by tracking ![][image6], ![][image3], and ![][image4] isotope compositions extracted from bone collagen.21

## **Data Quality and Reporting Standards for Stable Isotopes**

A highly functional standardisation tool must do more than simply rearrange column headers; it must evaluate the scientific validity of the data against established reporting standards. The archaeological and paleoecological communities have published extensive guidelines, notably through the BIAD (Big Interdisciplinary Archaeological Database) recommendations and the PAASTA (Palaeoproteomics And Archaeology, Society for Techniques and Advances) community, which draw heavily on the International Union of Pure and Applied Chemistry (IUPAC) protocols.22

### **Isotope Delta Scales and Reference Materials**

Legacy datasets frequently suffer from severe ambiguity regarding the reference scales used for stable isotope measurements. The standardisation tool must parse these datasets and ensure that delta values (![][image7]) are explicitly tethered to internationally recognised analytical standards.19

* **Carbon and Oxygen (![][image3], ![][image8]):** Must be reported relative to the Vienna Pee-Dee-Belemnite (VPDB) scale. For oxygen derived from water sources, the Vienna Standard Mean Ocean Water (VSMOW) scale is also frequently utilized, necessitating clear distinction in the metadata.19  
* **Nitrogen (![][image4]):** Must be reported relative to Atmospheric Nitrogen (AIR).19  
* **Hydrogen (![][image9]):** Must be reported relative to VSMOW.19  
* **Sulphur (![][image6]):** Must be reported relative to Vienna Cañon Diablo Troilite (VCDT).19  
* **Strontium (![][image5]):** Frequently utilized to trace prehistoric mobility patterns, strontium ratios must be reported alongside baseline environmental values to ensure comparability.26

### **Preservation Metrics and Diagenesis Indicators**

When processing bioarchaeological materials, particularly bone collagen, assessing diagenetic alteration is critical for determining data reliability. Legacy spreadsheets often contain raw elemental percentages (e.g., %C, %N, %S) and extracted yields.27 The standardisation tool must be programmed to automatically flag compromised samples based on community-accepted boundary conditions.27  
For standard bone collagen preservation, the accepted atomic C:N ratio typically falls between 2.9 and 3.6. Datasets reporting elemental weight %C:N values must be computationally converted to atomic C:N values for consistency across the repository.19 Furthermore, sulphur isotope analysis requires specific, taxonomy-dependent preservation bounds. In mammals and birds, acceptable collagen preservation dictates that sulphur concentration (%S) must range between 0.15% and 0.35%, the C/S ratio must fall between 300 and 900, and the N/S ratio must fall between 100 and 300\.28 For fish bone collagen, the parameters differ significantly: %S must range between 0.40% and 0.80%, C/S between 125 and 225, and N/S between 40 and 80\.28 An advanced standardisation pipeline should implement programmatic checks against these precise numerical thresholds, generating metadata flags for anomalous rows prior to repository submission.

## **Integration with Global Taxonomic Authorities**

Paleoecological datasets are inherently dependent on accurate taxonomic identifications. Because biological taxonomy is a living science subject to continuous revision, legacy datasets are highly susceptible to synonymous, outdated, or misspelled species names. To achieve cross-repository interoperability, the standardisation tool must programmatically query external taxonomic authority APIs to resolve arbitrary strings into permanent, globally recognised alphanumeric identifiers.

### **Global Biodiversity Information Facility (GBIF) Species Match API**

The GBIF taxonomic backbone represents one of the most comprehensive reference systems for biological nomenclature on the planet. The GBIF API features a dedicated species/match endpoint, which is heavily optimised for fuzzy matching against the GBIF backbone.29  
The primary RESTful endpoint operates via a simple GET request: https://api.gbif.org/v1/species/match?name={scientific\_name} 29  
Additional query parameters can be appended to narrow the algorithmic search space, such as kingdom=Plantae to restrict the taxonomic scope, or verbose=true to force the API to return multiple potential matches rather than just the most confident single result.31 Furthermore, version 2 of the species matching API supports identifier-based matching using fields like taxonID or scientificNameID. If an identifier and a scientific name are both provided, the API compares the matches and issues a flag in the response if they disagree.30  
The standardisation tool must parse the resulting JSON payload, which provides a highly structured taxonomic hierarchy:

| JSON Response Key | Description of Extracted Data |
| :---- | :---- |
| usageKey | The primary, permanent integer identifier for the taxon within the GBIF backbone (e.g., 5231190 for *Passer domesticus*).29 |
| matchType | Indicates the quality of the algorithmic alignment. Values include EXACT, FUZZY, or HIGHERRANK.29 |
| confidence | An integer score (0-100) indicating the probabilistic certainty of the match.29 |
| status | The taxonomic standing of the name, such as ACCEPTED or SYNONYM.29 |
| Hierarchy Keys | Provides the structural lineage via fields like kingdomKey, phylumKey, classKey, orderKey, familyKey, and genusKey.29 |

By extracting the usageKey and mapping it directly into the dataset's metadata, the standardisation tool ensures that the dataset remains irrevocably linked to the GBIF backbone, insulating the data against future nomenclature changes. Furthermore, the tool can leverage asynchronous download APIs via POST requests to pull massive occurrence datasets filtered by these taxon keys.32

### **Integrated Taxonomic Information System (ITIS) Solr Web Services**

For datasets primarily focused on North American species, or for analyses requiring an alternative governmental authority, the ITIS database provides a robust API ecosystem.33 Maintained by the USGS, ITIS houses data stretching back to 1977\.33 While ITIS maintains standard REST and SOAP endpoints, its Apache Solr web services offer highly flexible querying capabilities that are ideal for data wrangling.34  
The ITIS Solr endpoint is hosted at https://services.itis.gov/ and allows for complex search strings without requiring API keys.36 A standard query to retrieve JSON-formatted taxonomic data requires appending the wt=json parameter.37 The API supports searches by scientific name, vernacular names, and the Taxonomic Serial Number (TSN).36 Queries can be constructed using advanced Solr syntax, such as q=nameWInd:Homo\* to perform wildcard searches, returning highly nested JSON objects that detail the complete taxonomic hierarchy and associated synonymy.36

### **Marine and Aquatic Taxonomic Authorities (WoRMS and FishBase)**

Paleoecological analyses involving marine sediments, diatoms, or aquatic fauna require specialised registries. The World Register of Marine Species (WoRMS) offers a RESTful API designed specifically for marine taxa.39 The critical endpoint GET /AphiaRecordsByMatchNames utilizes the advanced TAXAMATCH fuzzy matching algorithm developed by Tony Rees.39 This algorithm is highly resilient to the morphological variants and typographic errors common in legacy marine spreadsheets. The output is anchored by the AphiaID, a unique identifier that serves as the gold standard for marine taxonomy.40  
For aquatic and fish-specific datasets, the FishBase and SeaLifeBase databases provide programmatic access. While early iterations relied on Ruby-based REST APIs, the modern architecture (e.g., rfishbase 4.0) has transitioned to distributing static data tables using the Parquet format, accessible via the HuggingFace datasets API and queried directly using tools like duckdbfs.43 This approach bypasses HTTP rate limits entirely, permitting massive, high-speed joins and taxonomic validations locally—a distinct advantage for a high-throughput standardisation tool.44

## **Machine Learning and NLP for Automated Schema Mapping**

The core mechanical challenge of standardising legacy datasets is mapping an endless variety of arbitrary column headers (e.g., "14C\_age\_BP", "Radiocarbon Date", "Age(yr)") to a rigid target schema like Neotoma's ndb.sampleages.age.4 Traditional rules-based matching engines and static lookup tables are inherently brittle, requiring constant manual maintenance. Therefore, the integration of Machine Learning (ML) and Natural Language Processing (NLP) into the standardisation tool is strictly necessary to achieve automation at scale.  
Automated data mapping leverages AI to detect schemas, apply semantic matching, and continuously monitor for schema drift across disparate datasets.45 Several open-source frameworks provide the necessary algorithmic infrastructure.

### **The Valentine Python Library**

Valentine is an extensible open-source Python library explicitly designed to evaluate and execute large-scale automated matching processes on tabular data (supporting both Pandas and Polars DataFrames).47 It operates via a unified API (valentine\_match), allowing developers to implement multiple foundational schema matching algorithms with minimal overhead.49

| Matching Algorithm | Mechanism and Approach | Best Use Case |
| :---- | :---- | :---- |
| **COMA (COmbining MAtch)** | A flexible approach that combines multiple complementary techniques, including linguistic matching based on column names/descriptions, structural matching based on schema hierarchies, and constraint-based matching.47 | Highly effective for standard datasets where column strings have basic structural similarity. |
| **Cupid** | A comprehensive schema matching algorithm that analyses structural hierarchies and linguistic features simultaneously.47 | Useful for mapping against deeply nested hierarchical relational schemas like Neotoma. |
| **DistributionBased** | Discovers relationships by analysing the statistical distribution of the instance data (the actual values in the rows) rather than relying solely on the column headers.47 | Critical for "messy" columns where headers are entirely missing, obfuscated, or poorly translated. |
| **SimilarityFlooding** | Constructs a graph representation of the schemas and casts schema matching as a complex graph-matching task.48 | Best utilized for complex, multi-table database mergers. |

By utilising a framework like Valentine, the standardisation tool can run an ensemble approach, combining the semantic understanding of column headers (COMA) with the statistical analysis of the underlying numeric data (DistributionBased) to generate high-confidence schema crosswalks automatically.

### **Deep Learning and Semantic Matching (SMAT)**

While traditional algorithmic matchers like COMA rely on structural and morphological heuristics, newer approaches employ deep learning NLP models. SMAT (Schema Matching using Attention over Attention) represents the vanguard of this methodology.50 SMAT operates exclusively on the attribute name and description, avoiding the need to encode domain-specific knowledge about the source or target systems.50 By leveraging pre-trained language models, SMAT understands the underlying semantics—recognising that terms like "profession" and "job", or in paleoecology, "depth" and "horizon", are semantically equivalent—surpassing the performance of traditional matchers by a wide margin.48 Integrating NLP embedding models (like those optionally supported in Valentine via pip install valentine\[embeddings\]) enables the standardisation tool to accurately decipher esoteric legacy abbreviations based on their contextual usage.49

## **Interoperability with PANGAEA and Broader Paleoclimate Repositories**

While Neotoma and IsoArcH serve specific, highly structured niches within paleoecology and archaeology, the standardisation tool must also account for broader interdisciplinary earth science repositories, most notably PANGAEA and the NOAA World Data Service for Paleoclimatology (WDS-Paleo).  
PANGAEA accepts a vast array of earth and life science data but operates under a distinctly different submission paradigm than the API-driven workflows of Neotoma. PANGAEA submissions require the author to adhere to strict Author Guidelines and proceed through a heavily mediated, human-in-the-loop editorial process.52 The communication, metadata provision, and file uploads occur primarily through the PANGAEA issue tracker rather than a direct programmatic endpoint.52 All submitted data and metadata are exhaustively quality-checked and harmonised for machine readability by the editorial team prior to the minting of a DOI; a process that can take weeks.52  
Because repositories like PANGAEA, NOAA WDS-Paleo, and Neotoma are increasingly integrated via federated search mechanisms—such as the Paleo Data Search API that allows users to query across proxy types globally using logical operators—duplicate submissions of the exact same dataset to multiple repositories are strongly discouraged.53 Therefore, the standardisation tool should allow the user to select a singular target repository schema and adapt its output format accordingly to prevent data fragmentation.  
In attempts to unify these disparate formats, community initiatives have proposed standards like LiPD (Linked Paleo Data) to serve as a universal data container for paleoclimatic data, facilitating the transfer of information among databases and into analytical environments like R.55 Bringing diverse groups like SISAL into Neotoma and LiPD enhances the visibility and accessibility of speleothem data while leveraging shared infrastructure.56 Ensuring the standardisation tool's output is compatible with Neotoma's tabular structure or LiPD's hierarchical structure will greatly enhance global data interoperability.10

## **Reproducible Data Transformation for FAIR Compliance**

The act of transforming a messy, legacy spreadsheet into a clean, repository-compliant dataset is intrinsically an act of data modification. Under the strictest interpretations of the FAIR principles, specifically Principle R1.2 (which dictates that metadata must include explicit provenance information about data creation or generation) 57, merely submitting the cleaned dataset is insufficient. The repository and end-users must have access to a machine-readable audit trail that documents exactly how the data was transformed from its raw state to its published state.58

### **The W3C PROV-O Ontology**

The gold standard for describing data provenance on the semantic web is the W3C PROV Ontology (PROV-O).60 PROV-O provides a domain-agnostic data model that defines provenance through three core classes and their interrelated properties 61:

1. **Entity (prov:Entity):** A physical, digital, or conceptual object. In the context of the standardisation tool, both the original legacy spreadsheet and the final harmonised CSV are modeled as discrete Entities.61  
2. **Activity (prov:Activity):** Something that occurs over time and acts upon or with entities. The execution of a Python transformation script, a unit conversion, or a Valentine schema match constitutes an Activity.61  
3. **Agent (prov:Agent):** A person, piece of software, or organisation that bears responsibility for an Activity or Entity.61

Transformational provenance is constructed as a Directed Acyclic Graph (DAG) using specific properties. When the standardisation tool modifies a dataset, it semantically records that the new Entity prov:wasDerivedFrom the old Entity.64 The generation of the new dataset is defined by the property prov:wasGeneratedBy linking it to the transformation Activity, which in turn prov:wasAssociatedWith the standardisation software itself (acting as the Agent).61 While PROV-O is highly robust and natively interoperable via RDF and JSON-LD serializations, its implementation requires significant architectural overhead to capture transformation logs at the atomic level, making it potentially too heavyweight for simple tabular scripts.61

### **Frictionless Data Packages and Declarative Pipelines**

A more pragmatic, structurally simple alternative for tabular data is the Frictionless Data framework. Frictionless Data focuses on JSON-encoded schemas where a datapackage.json acts as a manifest describing the structural elements of the dataset, including definitions of foreign keys, required data types, and missing value constraints.66 This approach treats metadata as a first-class citizen, ensuring that data typing and schema definitions remain rigidly intact during the transformation pipeline.69  
The datapackage-pipelines framework allows data transformations to be defined declaratively (often in a pipeline.yaml file) rather than imperatively.72 A Frictionless pipeline operates as a sequence of discrete processing steps that modify the descriptor, process the rows sequentially, and return output statistics.72 Powered under the hood by the lazy-loading capabilities of the PETL (Python ETL) library, Frictionless streams the data, enabling high-performance transformations of exceptionally large datasets without overwhelming local memory.69 By attaching the pipeline configuration file directly to the exported Data Package, the standardisation tool inherently provides the exact, reproducible logic used to standardise the data, satisfying FAIR provenance requirements with less structural complexity than a full RDF/PROV-O graph.60

### **Tool-Specific Provenance: OpenRefine, Trifacta, and pandas-profiling**

Data wrangling tools approach provenance tracking through diverse mechanisms, each offering distinct advantages. OpenRefine, a premier open-source tool for messy data, automatically records every atomic operation applied to a dataset within its internal workspace.json architecture.76 These operations are explicitly defined using the General Refine Expression Language (GREL). Users can extract this sequential history as a standalone JSON array and reapply it to identical or similar datasets, effectively creating a reproducible transformation script.76 When exporting an OpenRefine project archive (.tar.gz), the entire structural history is preserved, ensuring that the provenance is immutably linked to the dataset.78  
Comparatively, commercial data preparation platforms like Trifacta utilize proprietary "recipe" structures. A recipe in Trifacta acts similarly to an OpenRefine operation history but is compiled into highly optimised data processing engines (like Apache Spark or Dataflow) for execution at enterprise scale. While powerful, the proprietary nature of these recipes limits true open-science FAIR reproducibility compared to OpenRefine's JSON exports or Frictionless Data's open specifications.  
Conversely, tools like pandas-profiling (now ydata-profiling) do not inherently record sequential transformation steps. Instead, they generate comprehensive HTML or JSON reports detailing the *state* of the data (e.g., missingness matrices, variable distributions, correlation heatmaps) before and after transformation.  
Finally, the Statistical Data and Metadata eXchange (SDMX) standard, while prominent in macroeconomics and official statistics, focuses heavily on the structural transmission of aggregated data rather than the atomic, row-level provenance tracking required for standardising disparate paleoecological samples.  
For the proposed open-source standardisation tool, adopting a hybrid architecture is advisable. Capturing all automated schema mappings, regex replacements, and taxonomic resolutions into a sequential JSON audit log—akin to OpenRefine's GREL history or a Frictionless pipeline—provides the exact level of reproducible transformation documentation expected for FAIR-compliant submissions to rigorous repositories like Neotoma and PANGAEA.
